How This Website Works

From Cognito auth to DynamoDB leaderboard updates

By Ethan Barrett

Finance Clicker is a full-stack clicker game with server-authoritative scoring. The browser renders interactions instantly, but points, upgrade costs, passive-income accrual, and leaderboard writes are validated by Lambda before they are accepted. That separation is intentional: even if someone edits values in DevTools, persistent progression only changes when AWS confirms the request.

1) Authentication and Request Trust (AWS Cognito)

Sign-in is handled through Cognito Hosted UI and OAuth redirect flow. After login, the frontend stores the returned tokens in memory and sends the access token on API calls. API Gateway uses a Cognito authorizer so only verified identities can hit protected routes. Each Lambda receives claims from the token context and uses the user identity as the partition key, which ties all reads and writes to one authenticated player record.

2) Gameplay API Pipeline (API Gateway + Lambda)

Every meaningful action is sent through API Gateway to a dedicated Lambda route: /click, /buy, /trade, /state, and /leaderboard. This separation keeps each gameplay domain focused and auditable.

  • /click: Applies active-click earnings, then also settles idle income from elapsed time so players are credited even if they tab away.
  • /buy: Validates affordability for buildings/upgrades, applies scaling costs, increments ownership, and recomputes production stats that drive passive growth.
  • /trade: Executes market orders for buy, sell, and short-cover style flows using server-side checks for balance, quantity, and position integrity before committing portfolio changes.
  • /state: Returns canonical player state (currency, upgrades, portfolio, timestamped progression fields) so the UI can hydrate from trusted data.
  • /leaderboard: Reads score-facing data in a query-efficient format and sends a compact ranking payload back to clients.

3) Persistence and Consistency (DynamoDB)

DynamoDB stores long-lived player state (total points, upgrade tiers, passive rate, last update timestamp) and leaderboard-facing values. Lambdas read the current item, apply deterministic game rules, and write back the resulting state so clients do not become the source of truth. Because state is durable and keyed by verified identity, users can return on another browser and continue with the same progression while leaderboard reads remain fast and horizontally scalable.

4) Frontend Systems: Market UI, Candles, and Trade Simulation

The interface is designed like a live terminal: animated tape, layered panels, and constantly changing chart context. The candlestick panel is generated from synthetic OHLC sequences where each candle opens near the previous close, then computes high/low bounds and a close movement with controlled randomness. Volume bars are generated in the same loop so price action and volume feel correlated instead of arbitrary.

Trading simulation uses those generated prices as the market context for player decisions, but order outcomes are still validated server-side. Long positions track average entry versus current market price for unrealized gains, while short positions invert that relationship so gains occur when price drops and losses increase when price rises. The net result is game-like responsiveness in the browser with authoritative accounting in Lambda.

The ticker stream rotates through a broad symbol list with anti-repeat spacing rules so symbols do not appear in obvious loops. That keeps market motion feeling organic while remaining deterministic enough for consistent UI performance.

5) AWS Service Responsibilities

AWS services are split by responsibility: Cognito handles identity and token issuance; API Gateway terminates HTTP requests and enforces route authorization; Lambda hosts the game logic; DynamoDB stores progression, portfolio, and leaderboard-ready fields; CloudWatch captures logs/metrics for debugging and balancing. This architecture keeps the browser lightweight while allowing server rules to evolve without client-side trust.

Flow summary: Cognito verifies identity, API Gateway authorizes each route, Lambdas enforce clicker/trading economy rules (including /trade), and DynamoDB persists canonical state before the frontend re-renders.

7) Player Strategy Tips

  • Build a steady passive-income base first so every trade is backed by reliable cash flow.
  • Use smaller test positions before committing large capital to directional bets.
  • Short setups can pay off quickly, but they punish hesitation when momentum reverses.
  • Compare upgrade payback time, not just raw output—efficiency beats flashy purchases.
  • Check the leaderboard pace and set milestone targets to stay disciplined between sessions.
TL;DR: The frontend is interactive, but AWS remains authoritative—Cognito verifies users, API Gateway routes trusted calls, Lambdas apply scoring/trading logic, and DynamoDB persists progression plus leaderboard data.
