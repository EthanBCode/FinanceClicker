How This Website Works

From Cognito auth to DynamoDB leaderboard updates

portfolio breakdown

Finance Clicker is a full-stack clicker game with server-authoritative scoring. The browser renders interactions instantly, but points, upgrade costs, passive-income accrual, and leaderboard writes are validated by Lambda before they are accepted. That separation is intentional: even if someone edits values in DevTools, persistent progression only changes when AWS confirms the request.

1) Authentication and Request Trust (AWS Cognito)

Sign-in is handled through Cognito Hosted UI and OAuth redirect flow. After login, the frontend stores the returned tokens in memory and sends the access token on API calls. API Gateway uses a Cognito authorizer so only verified identities can hit protected routes. Each Lambda receives claims from the token context and uses the user identity as the partition key, which ties all reads and writes to one authenticated player record.

2) Gameplay API Pipeline (API Gateway + Lambda)

The game routes actions through /click, /buy, /state, and /leaderboard. The /click Lambda computes score increments from current points-per-click and elapsed idle time, then persists the updated totals. The /buy Lambda validates available points, applies exponential or tiered upgrade pricing, increments level counters, recalculates derived stats, and writes the new state in one controlled path. The /state Lambda returns a canonical snapshot for UI hydration, while /leaderboard performs a read-optimized query for top players and sends a compact payload for rendering.

3) Persistence and Consistency (DynamoDB)

DynamoDB stores long-lived player state (total points, upgrade tiers, passive rate, last update timestamp) and leaderboard-facing values. Lambdas read the current item, apply deterministic game rules, and write back the resulting state so clients do not become the source of truth. Because state is durable and keyed by verified identity, users can return on another browser and continue with the same progression while leaderboard reads remain fast and horizontally scalable.

4) Frontend Systems: Market UI, Randomized Charts, and Ticker Logic

The main UI keeps a trading-terminal feel with layered glass panels, animated telemetry, and procedurally generated market visuals. The candlestick chart is built from randomized OHLC data where each candle is derived from the prior close, then rendered with a matching histogram to represent synthetic volume. The ticker pulls symbols from a larger stock list and applies spacing guards so the same symbol is not emitted back-to-back or with only one symbol in between, which keeps the tape looking varied instead of repetitive. This docs page uses that same visual language but swaps the chart layer for a cinematic terminal stream to foreground the system architecture narrative.

Flow summary: Cognito verifies identity, API Gateway authorizes route access, Lambdas enforce game rules, and DynamoDB stores or returns state before API responses update the UI.

TL;DR: The frontend is interactive, but AWS remains authoritative—Cognito verifies users, API Gateway routes trusted calls, Lambdas apply scoring and economy logic, and DynamoDB persists progression plus leaderboard data.
